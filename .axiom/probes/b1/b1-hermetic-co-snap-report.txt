# Probe B1 ‚Äî hermetic CO SNAP regeneration report

**Plan:** `axiom-rebuild-plan-2026-07-02.md` ¬ß4-B1 (Phase B).
**Question B1 answers:** does the lockfile/regeneration model actually pay ‚Äî is
per-module regeneration cheap in tokens *and* reviewer-minutes, and are the diffs
against reviewed content dispositionable?
**Status:** measurement probe. No live statute/regulation/policy content was
changed. Regenerated output lives under `.axiom/probes/b1/`; lockfiles under
`.axiom/lockfiles/probe-b1/`.
**Path note:** the B1 spec named this report `docs/probes/b1-hermetic-co-snap-report.md`,
but `.axiom/repository-structure.yaml` (an enforced CI guard) disallows a top-level
`docs/` directory and disallows `.md` under `.axiom/**`. To keep CI green this
report is delivered as `.axiom/probes/b1/b1-hermetic-co-snap-report.txt` (markdown
content, `.txt` extension) and the lockfile README as `README.txt`.

## TL;DR tripwire verdict

**Regeneration is CHEAP in dollars/tokens but NOT dispositionable-cheap, and the
"encodings are cache" premise does not hold at this model quality/config.**
Of 25 modules, **1 regenerated byte-identical, 1 was mixed (one new-correct
addition bundled with an old-correct regression), and 23 were old-correct** ‚Äî
i.e. the reviewed content wins and the regenerated module would regress or break
consumers if applied. The dominant failure mode is **public-name instability**
(invariant 6): for state regulation modules the regenerator discards the reviewed
rule set wholesale and invents new output names, because it cannot recover the
established naming from source text alone. Token cost was trivial ($1.91 for 25
modules); the real cost is reviewer attention, and every non-identical module
needed a human. Per plan ¬ß4-B1: **keep lockfiles as provenance-only; do NOT plan
broad regeneration on the current stack.**

## Honest deviations (read first ‚Äî they bound every number here)

1. **Model.** The configured default `codex:gpt-5.5` was non-runnable: the codex
   CLI's vendored binary is missing (ENOENT), `gpt-5.5` is not exposed by the
   available OpenAI key (max is `gpt-5.2`), and the `claude` backend 401s from the
   nested CLI in this environment. Regeneration executed on **`openai:gpt-5.2`**
   via the OpenAI Responses API (reasoning effort `low`, `max_output_tokens`
   16384) ‚Äî the closest available frontier OpenAI model. Every token/diff number
   is a gpt-5.2 measurement. The *direction* of the verdict (name instability,
   old-correct dominance) is a structural property of prompting from source text
   without the naming registry, and would likely persist under gpt-5.5, but the
   absolute disposition split should be re-measured under the true default before
   any consolidation decision leans on it.
2. **Oracle gates could not run.** `policyengine_us` is not installed in any local
   venv, so (a) the per-module PolicyEngine oracle returned `None` for all 25
   modules (including the 5 PE-mapped ones), and (b) the population-level
   `snap-populace-compare --jurisdiction us-co` could not be executed live.
   Dispositioning therefore uses **content diff + companion-test semantics + the
   committed CO SNAP population baseline** (dashboard report
   `axiom-policyengine-co-snap-ecps.json`: **730 CO households √ó 2 concepts =
   1,460 comparisons, 100% Axiom‚ÜîPolicyEngine match, 0 mismatches** ‚Äî the bar the
   live modules currently clear). Any regenerated module that would break that
   baseline is old-correct.
3. **CI is not a usable per-module signal in this config.** In `repo-augmented`
   mode the harness copies the *live* target module into the workspace context.
   CI then sees the regenerated rule as a **duplicate of an "upstream" target**
   (the live copy) and rejects it as a placement violation; the generalist
   reviewer trips on the same thing. Result: `ci_pass=0/25` and
   `generalist_review_pass=0/25` are harness artifacts, not evidence about
   content. This is itself a finding: hermetic regenerate-in-place needs the live
   target excluded from its own context, or the duplicate-target guard relaxed for
   probe runs.
4. **r0 provenance is partial on main.** The r0 release manifest (corpus) and the
   40,991-concept registry (corpus#171) are open PRs, not merged. Lockfiles pin
   the r0 corpus *git commit* `9ee50de` (merged) + the r0 manifest sha, and the
   *encode-side* concept seed the pipeline actually consumes. The A1 JSON Schemas
   are not on the engine HEAD used here, so `schema_version` pins the engine
   package version + git commit that compiled/validated the modules.

## Probe set (25 modules) and selection

The probe set is the modules exercised by axiom-encode's CO SNAP benchmark lanes
(`benchmarks/us_snap_*` with CO/SNAP cases) **plus** the core of the `us-co` SNAP
composition chain (`us-co/policies/cdhs/snap/fy-2026-benefit-calculation.yaml`,
which imports ~52 modules), capped at 25. Priority: (1) the 8 benchmark-lane
modules with a PolicyEngine oracle hint; (2) the deepest/most load-bearing federal
and CO-regulation modules the composition imports; (3) `chain-backfill` CO modules
to reach 25 after excluding source-absent imports (below). Full machine-readable
set: `.axiom/probes/b1/probe-set.json`; per-module rationale in each lockfile's
`selection_rationale`.

**Hermeticity gap found while scoping (decision-relevant):** 8 modules the CO
SNAP composition imports **cannot be regenerated at r0 because their corpus source
is absent from the pinned release** ‚Äî the entire **7 CFR 273 federal family**
(`273/3,4,5,6,9,10,24`) and the FY2026 income-eligibility guidance `page-3`. These
were replaced with resolvable CO composition modules. That the flagship program's
composition depends on 7 federal modules with no r0 corpus source is a concrete
prerequisite the A2/A5 workstreams must close before the full composition is
hermetically regenerable at all.

## Why 23 are old-correct ‚Äî the mechanism

- **Public-name instability (invariant 6) is the dominant failure.** For CO
  regulation modules the regenerator produces a rule set with **near-zero overlap**
  with the reviewed public output names:
  - n=8 (`4.407.5`, child support): live has 1 rule `child_support_deduction`
    (the name the composition + companion test consume); regen replaced it with 3
    differently-named rules and dropped `child_support_deduction`.
  - n=9 (`4.403.11`, self-employment): live's 7 `snap_`-prefixed rules ‚Üí a disjoint
    4-rule set, zero overlap.
  - n=10 (`4.405`, income exclusions): live's 21 rules ‚Üí 16 new names, zero overlap
    (the model even embedded the section number into rule names, e.g.
    `..._excluded_amount_under_4_405`).
  The benchmark's `policyengine_rule_hint` *steers* the primary output name, which
  pushes the regenerated module further from the reviewed naming. Applying any of
  these would break `finbot` input schemas, microsim slot names, oracle
  `concept_mappings`, and claims subjects ‚Äî exactly the consumer surface invariant
  6 protects.
- **Dropped cross-reference handling.** n=1 (`e/2/B`): the live module carries a
  `deferred_outputs` block deferring `snap_earned_income_deduction` because the
  subparagraph (C) exception source is not in-workspace; the regen omitted the
  deferral entirely (old-correct).
- **Input-reference / formula drift.** n=3 (input var swap), n=2 (formula delta) ‚Äî
  reviewed variable choices win.
- **Guidance-page source-id modules regenerate to empty stubs.** n=6 and n=7
  (federal FY2026 deductions and income-eligibility tables) came back with
  `rules: []` and `status: upstream_source_check_required` at a *page-keyed path*,
  reproducing none of the live 9‚Äì12 reviewed table rules. The model's caution is
  defensible, but it yields zero usable content vs. the reviewed module.
- **The one clean win:** n=5 (`maximum-allotments`) regenerated **byte-identical**
  ‚Äî a well-scoped, single-source policy table with stable names is the case where
  "encodings as cache" does hold.

## What did NOT discriminate (and why the verdict rests on content diff)

The three automated gates the lockfile model would lean on all failed to yield a
per-module disposition on the current stack:
- **CI**: 0/25 ‚Äî duplicate-target placement violation from the live copy in context
  (harness confound), not content.
- **Generalist reviewer**: 0/25 ‚Äî same root cause.
- **PolicyEngine oracle** (per-module and population): 0/25 ran ‚Äî `policyengine_us`
  absent. Note that only **5 of 25** modules even have a PE mapping edge; the other
  20 have `oracle: none` and are structurally invisible to the PE oracle even when
  it runs ‚Äî consistent with the plan's ¬ß1 "oracle covers ~1/3" finding.

## Reviewer-cost reading (the number Phase C needs)

125 reviewer-minutes for 25 modules = **~5 min/module of confident dispositioning**
once the failure pattern was known. But that undercounts real cost two ways: (a)
the disjoint-rule CO modules were *quick to reject* precisely because they all fail
identically ‚Äî a reviewer seeing them fresh, or trying to **reconcile** rather than
reject (map new names back to reviewed ones, verify each formula), would spend
materially more; (b) with the PE oracle dark, a conscientious reviewer would have
to hand-check numeric equivalence that the oracle should automate. Extrapolated
honestly: the plan's B1 tripwire of ">10 reviewer-days or majority-undispositionable
diffs" for full CO SNAP is **plausibly reached** on the reconcile interpretation ‚Äî
the diffs are dispositionable, but they are overwhelmingly *rejections*, and the
value of regeneration (finding cases where new beats old) showed up in exactly
**1 of 25** modules, itself impure.

## Tripwire verdict (per plan ¬ß4-B1)

- **Cheap in compute?** Yes ‚Äî $1.91 / 25 modules, 20 min wall. Not a constraint.
- **Cheap + dispositionable in review?** **No.** 23/25 old-correct, 1/25 mixed,
  1/25 identical, 0/25 clean new-correct. The premise that "encodings are a cache
  the model can cheaply reproduce" fails here: absent the naming registry and a
  live oracle, regeneration reliably reinvents public names and must be rejected.
- **Recommendation:** treat probe-b1 lockfiles as **provenance-only** and **do not
  plan broad regeneration** on the current stack/model. Two concrete unlocks the
  data points to before B1 could flip positive: (1) feed the concept/name registry
  (A3) into the encoder so it reuses reviewed output names instead of inventing
  them ‚Äî this alone would likely convert many old-correct rejections into
  equivalents; (2) make the per-module + population PolicyEngine oracle runnable in
  the regeneration harness (install pe-us; exclude the live target from its own
  context) so dispositions are oracle-backed, not hand-diffed. Re-run B1 under the
  true default model after both.
